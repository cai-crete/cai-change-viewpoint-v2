## 1. 현황 분석 — 비율 찌그러짐의 근본 원인

### 1-A. 원인 체인

```
AI 출력 이미지:
  gemini-3.1-flash-image-preview → 정사각형 고정 출력 (예: 1024×1024px)
  출력 해상도 제어 불가 (API 스펙 제한)

3×3 등분 크롭:
  cellW = img.width  / 3  → 341px (정사각형)
  cellH = img.height / 3  → 341px (정사각형)

건물 실제 입면 비율 (예: W=8, D=6, H=15):
  FRONT: W × H = 8 × 15 → 가로:세로 = 1:1.875 (세로형)
  LEFT:  D × H = 6 × 15 → 가로:세로 = 1:2.5  (더 세로형)
  TOP:   W × D = 8 × 6  → 가로:세로 = 1:0.75 (가로형)
  각 셀: 341 × 341       → 가로:세로 = 1:1   (정사각형)

결과:
  AI가 세로형 입면을 정사각형 셀에 억지로 채움 → 가로 방향 압축(찌그러짐) 발생
  trimAndCenter()는 비율 왜곡 자체를 복원 불가
  → Phase 4 최종 생성 시 왜곡된 입면이 기준 이미지로 주입 → 전혀 다른 건물 출력
```

### 1-B. 사용자 제안 ("제한 없는 규격에서 생성 후 크롭") 평가

| 항목 | 평가 |
|---|---|
| 출력 해상도 증가 가능 여부 | ✗ 불가 (Gemini API 출력 해상도 고정) |
| 고해상도 이미지 생성 가능 여부 | ✗ 불가 (prompt로 제어 불가) |
| 크롭 전 전체 이미지 완성도 | △ 현재도 동일한 방식 |
| 비율 왜곡 해결 여부 | ✗ 정사각형 셀 문제는 동일하게 유지 |

**결론**: 정사각형 출력 고정 제약이 존재하는 한, 단일 3×3 이미지 생성 방식으로는 비율 왜곡 해결 불가.

## 2. 해결 방안 비교

### 방안 A — 패널별 개별 생성 (권장)

**개념**: 5개 입면을 각각 별도 API 호출로 생성. 각 호출에 올바른 비율 지시.

```
API 호출 5회 (Promise.all로 병렬 실행):
  call 1: FRONT elevation → "Width:Height = W:H ratio, orthographic, transparent bg"
  call 2: REAR  elevation → "Width:Height = W:H ratio, 180° rotated"
  call 3: LEFT  elevation → "Width:Height = D:H ratio, left side view"
  call 4: RIGHT elevation → "Width:Height = D:H ratio, right side view"
  call 5: TOP   elevation → "Width:Height = W:D ratio, roof plan"

각 호출 입력:
  - IMAGE 1 (원본 사진)
  - 해당 입면의 AEPL 파라미터 (Phase 1 결과)
  - 명시적 비율 지시 (W:H 또는 D:H 또는 W:D)
```

**결과**:
- 각 이미지가 자체 비율로 생성됨 (정사각형 셀 제약 없음)
- AI가 각 면을 독립적으로 집중 생성 → 정확도 향상
- 왜곡 없이 올바른 비율의 입면 이미지 획득

**장단점**:
| 장점 | 단점 |
|---|---|
| 비율 왜곡 근본 해결 ✓ | API 호출 5회 (비용 5배) |
| 각 입면 집중 생성 → 품질 향상 ✓ | Promise.all 병렬 실행 시 총 시간은 1회와 유사 |
| 크롭 불필요 (직접 사용 가능) ✓ | 입면 간 교차 참조 없음 |
| bldgRatio 비율 정확히 반영 ✓ | — |

---

### 방안 B — 비등분 크롭 (bldgRatio 기반)

**개념**: 단일 3×3 이미지 유지. 크롭 경계를 bldgRatio 비율로 계산.

```
현재 크롭: cellW = img.width / 3 (등분)

변경 후 크롭:
  unitsW = 2*H + W,  unitsH = 2*H + D
  x0 = img.width  * (H / unitsW)         // LEFT-CENTER 경계
  x1 = img.width  * ((H + W) / unitsW)   // CENTER-RIGHT 경계
  y0 = img.height * (H / unitsH)         // REAR-MIDDLE 경계
  y1 = img.height * ((H + D) / unitsH)   // MIDDLE-FRONT 경계

각 패널 크롭:
  rear:  (x0, 0,  x1, y0)
  left:  (0,  y0, x0, y1)
  top:   (x0, y0, x1, y1)
  right: (x1, y0, img.width, y1)
  front: (x0, y1, x1, img.height)
```

**장단점**:
| 장점 | 단점 |
|---|---|
| API 호출 1회 유지 (비용 동일) | AI가 비등분 배치를 정확히 따르지 않을 수 있음 |
| 구현 범위 작음 | 정사각형 출력 내 왜곡 자체는 여전히 존재 |
| 빠른 적용 가능 | bldgRatio 데이터를 generateSitePlan에 전달 필요 |

#### A-2. `generateSingleElevation()` 신규 함수

```typescript
const generateSingleElevation = async (
  base64Image: string,
  panel: { id: string; label: string; wUnit: number; hUnit: number },
  aepl: any,
  bldgRatio: { width: number; depth: number; height: number }
): Promise<string> => {
  const prompt = `
# Architectural Elevation Generator — Single View

## Target View: ${panel.label}
- Width : Height ratio = ${panel.wUnit} : ${panel.hUnit}
- Projection: Orthographic (no perspective)
- Background: 100% transparent (alpha channel mandatory)
- No text, no labels, no shadows beyond the building

## Constraints
- IMMUTABLE: Building geometry, proportions, materials from IMAGE 1
- Render ONLY the ${panel.label} face of the building
- Fill the output image to the ${panel.wUnit}:${panel.hUnit} aspect ratio
- Ground line at the bottom of the image

## AEPL Reference
${buildAEPLForPanel(aepl, panel.label)}
  `.trim();

  const result = await ai.models.generateContent({
    model: IMAGE_GEN,
    contents: { parts: [
      { inlineData: { data: base64Image.split(',')[1], mimeType: base64Image.split(';')[0].split(':')[1] } },
      { text: prompt }
    ]}
  });
  // 반환: 이미지 data URL
  ...
};
```
