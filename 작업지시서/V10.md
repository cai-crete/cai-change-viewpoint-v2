## 1. 이미지 재생성이 PHASE 2에 미치는 효과

### PHASE 2 실제 입력값 추적

```
generateSitePlan(base64Image, extractedParams)
  ↓
contextualParamsStr = JSON.stringify(extractedParams)
  ↓
sitePlanPrompt: "...matching the established context and the following extracted parameters: ${contextualParamsStr}"
  ↓
gemini-3.1-flash-image-preview 호출
```

**결론: 이미지 재생성이 PHASE 2에 미치는 효과는 미미합니다.**
- PHASE 2는 원본 `base64Image`를 직접 받음 → 재생성 이미지는 PHASE 2에 전달되지 않음 (제안 3 방식)
- `contextualParamsStr` = `JSON.stringify(extractedParams)` — 구조화되지 않은 날(raw) JSON 덤프를 그대로 전달
- → PHASE 2 AI는 정형화된 파라미터 없이 이미지만으로 5면을 추론해야 함

---

## 2. PHASE 2 정확도 향상 방법 3가지

### 방법 A. `contextualParamsStr` 구조화 포맷 주입 ★ (가장 효과적)
**현재:** `JSON.stringify(elevationViews)` — 원시 JSON 전체를 텍스트로 전달  
**변경:** 5-View AEPL 스키마에서 핵심값을 추출해 사람이 읽을 수 있는 포맷으로 명시

```
[FRONT] Material: Brick Red, Glazing: 40%, Grid: 3.6m module, Floors: 10
[REAR]  Type: Service facade, Small ventilation windows, MEP on roof
[LEFT/RIGHT] Wrap front material, minimize openings (Type A Urban)
```
→ AI가 실제로 이해하는 자연어 기반 파라미터로 5면 생성 지시 가능.  
→ 구현 난이도 낮음. 기존 5-View AEPL 데이터(V9에서 이미 추출)를 재활용.

### 방법 B. 재생성 이미지를 PHASE 2에도 전달 (Proposal 3 확장)
PHASE 2 `generateSitePlan()` 호출 시에도 내부적으로 regenerated 이미지를 사용.  
→ AI-Normalized 이미지를 기반으로 전개도 생성 → 왜곡·노이즈 없는 입력.  
→ 구현 난이도 중간. 업로드 이미지 품질이 낮을수록 효과 큼.

### 방법 C. 시점 정보(Angle/Altitude/Lens) 명시적 주입
현재 sitePlanPrompt에는 분석된 시점 정보(06:00, 200m, 35mm)가 전달되지 않음.  
→ "Source image is taken from 07:30, 200m altitude, 35mm lens" 를 명시하면 AI가 원본 시점과 다른 5면을 더 논리적으로 생성 가능.  
→ 구현 난이도 낮음. 기존 `analyzedOpticalParams` 값 재사용.
