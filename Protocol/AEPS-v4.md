# AEPS-v4

# 프로토콜 개요
- 3DS 오브제로 변환 가능한 명확한 5면 전개도(Front, Right, Left, Top, Back)의 동시 출력이라는 궁극적 목표를 달성하기 위한 마스터 프로토콜입니다.
- 파이프라인 전반을 통제하는 데이터 정제 원칙(철학)과 렌더링 실행 규격(코드)을 하나로 통합하여 작동합니다.

## 1. BIM 기반 데이터 철학 (The Revit Paradigm)
- 아키텍처 차용: 본 시스템은 Revit과 같은 상용 BIM 소프트웨어의 데이터 아키텍처를 시스템 레벨에서 도입하여 사용합니다.
- 객체의 삼위일체: 물리적 실체인 벽(Wall)을 단순한 선과 면의 집합이 아닌, 형태(Geometry), 속성(Properties), 관계(Relations)의 결합체로 정의한 것과 같이 건축물 정보(객체)는 삼위일체로 구성됩니다.
- 파라미터(Parameter)의 역할: 객체가 시스템 및 외부 세계와 소통하기 위해 사용하는 명확한 어휘(Vocabulary)로 기능합니다.
- 데이터 해석 기준: 입력된 이미지를 추상적 픽셀이 아닌, 물리적 좌표를 지닌 객체와 그 객체를 구체적인 수치로 증명하는 속성으로 구조화하여 다룹니다.

## 2. 모달리티의 상호보완적 1:1 매칭 (Image Prompt ↔ Text Prompt)
### 2-1. ensemble_pair 원칙 (관계 정의)
- 형태(Geometry)와 속성(Property)은 각자의 역할 영역을 명확히 구분하되, Image Prompt ↔ Text Prompt의 `ensemble_pair`로 항상 쌍을 이루며 하나의 완전한 객체 정보(Object)로 통합·귀결된다.
- 단독 유효성 규칙:
  * Geometry 없는 Property = 귀속 대상 없는 부유 데이터 → 무효
  * Property 없는 Geometry = 외곽선만 존재하는 빈 껍데기 → 불완전
  * `ensemble_pair` 상태에서만 → 완전한 객체 정보 성립
- 정보 구분의 목적:
  * 형태와 속성을 나누는 것은 단순한 분리가 아니라, 완벽한 상호보완과 합의를 이끌어내기 위한 역할 구분입니다.
  * 직관적인 시각 정보(Image Prompt)로 형태의 틀을 확정하고, 정제된 파라미터 정보(Text Prompt)로 그 틀에 정확한 치수와 물성을 귀속시킵니다.

### 2-2. 1_Geometry_Data (시각적 합의와 공간의 규정)
- `ensemble_pair` 내 역할: 객체의 시각적 외곽선과 공간 좌표를 확정하는 **형태 확정자(Shape Anchor)**로 기능합니다.
- 모달리티: 오직 Image Prompt(Z-Depth 및 Normal Map)로만 변환됩니다.
- 출력 기준:
  * 사각형, 원형 등 직관적인 형태(Shape)와 3D 공간 상의 위치(X, Y, Z)를 시각적으로 표현합니다.
  * 픽셀이 존재해야 할 절대적인 외곽선(Outline)과 바운딩 박스를 확정합니다.
- 단독 상태: Property와 `ensemble_pair`를 이루지 않은 Geometry는 불완전 객체로 간주하며, 다음 Phase로 이관하지 않습니다.

### 2-3. 2_Property_Data (구체적 정보값의 수치화)
- `ensemble_pair` 내 역할: 확정된 형태의 경계 내부를 수치와 물성으로 빈틈없이 채우는 **정보 귀속자(Data Binder)**로 기능합니다.
- 모달리티: 오직 Text Prompt(파라미터 문자열)로만 작동합니다.
- 출력 기준:
  * 확정된 Geometry에 특정 치수(예: W1000×H1000) 및 재질(예: Low-E 유리)과 같은 구체적 정보값을 주입합니다.
  * Geometry가 규정한 시각적 경계를 설명하고 증명하기 위해 존재합니다.
- 단독 상태: Geometry와 `ensemble_pair`를 이루지 않은 Property는 부유 데이터로 간주하며, 다음 Phase로 이관하지 않습니다.

### 2-4. ensemble_pair 검증 게이트 (Phase 간 이관 조건)
- 매 Phase 종료 시점에 아래 조건을 충족한 객체만 다음 단계로 이관됩니다.

## 3. 데이터 정제 및 인수인계 사이클 (Data Lifecycle)
- 사이클 개요: 시각적 그릇(Geometry)과 정보적 채움재(Property)의 상호보완 구조는 Protocol A의 매 단계에서 작동하며, Protocol B로 이관되는 완벽한 핸드오버(Handover) 규격이 됩니다.
- "Protocol A: Architectural Logic Engine (건축 계획 엔진)"의 프로세스에 따른다.
```
- Phase 1 : Visual Diagnosis & Element Extraction (가시권 진단 및 건축 요소 추출)
- Phase 2 : Context Inference & Zoning (컨텍스트 및 조닝 추론)
- Phase 3 : Blind Spot Reconstruction Strategy (비가시권 재구성 전략)
- Phase 4 : Design Specification Output (건축 설계 제원 산출 및 데이터 패키징)
```

# [Data Handover Format: AEPL Schema]

- 범용 마스터 표준: Protocol A(건축 계획 엔진)에서 정제된 데이터를 Protocol B(시각화 실행 엔진)로 안정적으로 인수인계하기 위한 규격
- 앙상블 페어(ensemble_pair) 맵핑: 형태(Geometry)와 속성(Property)은 서로의 영역을 침범하지 않으면서도 완벽한 상호보완적 앙상블 페어로 결합하여 하나의 온전한 건축 실체를 완성함
- 1_Geometry_Data_MASTER (형태 확정자): 렌더링 시 이미지 프롬프트(ControlNet 가이드)로만 변환되어, 픽셀이 존재해야 할 형태의 시각적 경계와 3D 공간 좌표를 단단하게 고정함
- 2_Property_Data_SLAVE (정보 귀속자): 렌더링 시 텍스트 프롬프트로만 변환되며, 이미지 프롬프트가 확립한 시각적 틀 내부에 치수, 재질, 광학적 수치 등 구체적인 정보값을 오차 없이 타설함

# [AEPL 마스터 스키마]

- 1. 프로젝트 메타데이터 및 전역 제약 조건
  * 버전 및 템플릿: AEPS-v4 패러다임을 기반으로 하는 범용 건축 매스 추출기(Universal Architectural Mass Extractor)
  * 타겟 객체 및 시스템 목표: 건축물 매스 마스터를 대상으로 하며, 최종적으로 5면 정사영 전개도를 동시 출력하는 것을 목표로 함
  * 전역 제약 조건 (Project Global Constraints): 외부 환경 및 도시적 맥락을 포함하는 건축물 렌더링을 실행. 이를 위해 별도로 제공되는 도시 맥락 파라미터를 통합 적용하여 건축물 주변의 환경을 구성, 대상 건축물과 도시 맥락이 자연스럽게 안착되도록 입체감 확보를 위한 동적 환경광 차폐(Dynamic AO) 기본 강도를 설정.

- 2. 뷰 시퀀스 (View Sequence)
  * 출력 순서: 정면, 평면/탑, 우측면, 좌측면, 배면 순으로 총 5개의 뷰를 자동 순회(Auto Iterate)함
  * 명명 규칙 및 완료 조건: 각 객체는 지정된 외벽 명명 규칙에 따라 고유하게 식별되며, 5개의 뷰가 모두 렌더링되어 앙상블 페어 조립이 완료되면 파이프라인 트리거가 종료됨

### **3. 뷰별 절대 카메라 및 법선 벡터 매핑 (View Camera Map)**
각 뷰포인트는 직교 투영을 위해 절대적인 방위각, 고도, 3D 좌표계 상의 법선 벡터를 가집니다.
* **정면 (Front):** 카메라 방위각 0, 고도 0 / 법선 벡터 (0, -1, 0) / 역할: Primary_Facade
* **평면/탑 (Top):** 카메라 방위각 0, 고도 90 / 법선 벡터 (0, 0, 1) / 역할: Top_Elevation
* **우측면 (Right):** 카메라 방위각 90, 고도 0 / 법선 벡터 (1, 0, 0) / 역할: Right_Facade
* **좌측면 (Left):** 카메라 방위각 270, 고도 0 / 법선 벡터 (-1, 0, 0) / 역할: Left_Facade
* **배면 (Back):** 카메라 방위각 180, 고도 0 / 법선 벡터 (0, 1, 0) / 역할: Rear_Facade

### 4. 공통 파라미터 딕셔너리 (Shared Entity Defaults)
- 파라미터 저장 기준: 각 단계에서 정제된 모든 건축적 분석 정보는 형태(Geometry)와 속성(Property)이 상호보완적으로 결합된 {Image Prompt ↔ Text Prompt} 앙상블 페어(ensemble_pair) 형태로 항상 짝을 이루어 저장됨.

- A. 1_Geometry_Data_MASTER (시각적 형태 및 객체 데이터)
  * 역할: [ABSOLUTE MASTER] 오직 3D 화이트박스 구축 및 이미지 프롬프트(Z-Depth/Normal)로만 변환되어 시각적 합의를 이루는 앙상블 페어의 형태 확정자. 정면(Front Facade)을 6시 방향으로 앵커링한 절대 기준 좌표를 따름.
  * 매스 및 한계 체적: 입면의 폭, 층고 및 건물 전체 절대 높이, 매스의 분절과 셋백, 캔틸레버 등 3D 기하학적 비례와 최대 허용 체적 규정
  * 코어 및 평면 구조: 수직 동선(계단/승강기) 및 기계 설비 샤프트의 X, Y 절대 좌표, 용도별 내부 조닝 경계 확정
  * 개구부 및 입면 분절: 창면적비(0.0~1.0 범위), 창호의 함몰(음수) 및 돌출(양수) 깊이, 커튼월 멀리언 그리드 간격, 수직/수평 루버 및 발코니 배열 모듈 
  * 지붕 및 기능적 객체: 평지붕/경사지붕 구조, 파라펫 높이, 배수로, 상층부 기계설비(HVAC) 및 유지보수 장비의 절대 좌표계 위치값 배치
  * 대지 및 도시 맥락 기하학: 인접 건물 매스의 스케일 및 이격 거리, 보차도 분리 경계선, 조경 마운딩 및 외부 계단의 물리적 구조선 투영

- B. 2_Property_Data_SLAVE (정보적 속성 데이터)
  * 역할: [SUBORDINATE SLAVE] 형태를 치수(dimension)로 기재, 오직 텍스트 프롬프트로 변환되어 기하학적 경계 내부를 구체적인 수치로 채우는 앙상블 페어의 정보 귀속자
  * 표면 물성 및 디테일: 노출 콘크리트, 금속 패널 등 외장재의 PBR 속성(거칠기, 금속성, 반사율), 베이스 고유 색상(Hex 코드), 마감재 패널의 질감 및 줄눈의 너비/깊이 
  * 광학 수치 및 유리 속성: 투명도, 굴절률(IOR), 빛 투과율, 로이(Low-E) 코팅 색상 등 창호 및 마감재의 광학적 특성 명시
  * 환경적 현실성 및 에이징: 시간 경과에 따른 풍화 레벨, 백화 현상, 오염 궤적, 물때 등 표면의 세월감 및 리얼리티 제어
  * 시간, 조명 및 미세 기후: 구체적 시간대 및 계절에 따른 태양 방위각/고도, 그림자 선명도(AO/GI), 대기 산란 투과도, 야간 파사드 조명(색온도, 투광 조명 등) 및 젖은 노면과 같은 동적 디지털 트윈 상태 변수 반영

### 5. 개별 객체 조립 (Entity_List) 및 앙상블 페어 결합
- 뷰 시퀀스 배열: 자동 순회 순서에 따라 정면, 평면/탑, 우측면, 좌측면, 배면 등 5개의 뷰가 독립적 객체로 자동 배열됨
- 속성 상속: 각 뷰의 객체는 공통 파라미터 딕셔너리의 기본값을 일관되게 상속받음
- 앙상블 페어 데이터 패키징: Protocol A가 대지 환경 및 건물 구조를 심층 추론하여 도출한 각 입면의 고유한 공간 좌표와 치수는 시각적 뼈대인 1_Geometry_Data_MASTER(Image Prompt)에 매핑되고, 표면 재질, 환경 조명, 에이징 상태 등의 수치는 해당 외벽 경계 내의 2_Property_Data_SLAVE(Text Prompt)에 덧씌워짐. 이를 통해 두 모달리티가 완벽하게 일대일 매칭되는 {형태(Geometry) ↔ 속성(Property)} 앙상블 페어(ensemble_pair) 구조의 최종 데이터 블록으로 완성되어 시각화 엔진으로 이관됨.
