# 작업지시서 V6

**작성일:** 2026-04-02
**목적:** 새로고침 후에도 캔버스 이미지가 유지되도록 상태 영구화(Persistence) 시스템 도입

## 1. 현황 분석

현재 `App.tsx`는 모든 캔버스 아이템(`canvasItems`)을 오직 React `useState` 메모리에만 저장합니다. **현재 어떠한 영구 저장 메커니즘도 없어, 새로고침 시 서버로부터 빈 화면만 로드**됩니다.

보존이 필요한 데이터:
- `canvasItems[]`: 업로드된 원본 이미지(base64), X/Y 좌표, 크기, parameters(생성된 전개도 등)
- `canvasOffset`, `canvasZoom`: 화면 뷰포트 위치/줌 상태

> [!IMPORTANT]
> `canvasItems`에 포함된 이미지(base64)는 이미지 1장당 수 MB에 달할 수 있어 저장 용량이 핵심 제약 조건입니다.

---

## 2. ⚠️ 판단 대기 및 3가지 제안

원하시는 저장 방식 **1가지**를 승인해 주시면 해당 방식으로 구현하겠습니다.

**제안 1. IndexedDB 방식 (다용량, 가장 강력)** ← **[승인 완료 - 구현 적용]**
- **저장 위치:** 브라우저 내 `IndexedDB` (`cai-canvas-db` 스토어)
- **자동 저장:** `canvasItems`, `canvasZoom`, `canvasOffset` 변경 시 800ms 디바운스 후 자동 저장.
- **자동 복원:** 앱 첫 마운트 시 `dbLoad()`로 이전 상태를 전부 읽어 캔버스를 완벽 복원.
- **개발자 확인:** `F12` 브라우저 콘솔에 `[IndexedDB] Canvas state auto-saved.` 및 복원 메시지 표시.
- **저장 위치:** 브라우저 내 `IndexedDB` (수백MB ~ GB 수준 대용량 지원)
- **동작 방식:** `canvasItems` 배열이 변경될 때마다 자동으로 IndexedDB에 저장. 새로고침 시 저장된 데이터를 읽어 캔버스를 완벽하게 복원.
- **복잡도:** 구현이 가장 복잡하지만, 이미지 base64 포함 대용량 데이터를 안정적으로 다루기에 가장 적합.
- **권장도:** ★★★ (대용량 이미지 처리에 최적)

### 제안 2. localStorage 방식 (소용량, 가장 빠름)
- **저장 위치:** 브라우저 `localStorage` (최대 5~10MB 한계)
- **동작 방식:** 이미지 데이터 자체가 아닌, 파일 메타(좌표, 파라미터)만 저장하고 이미지는 세션 간 복원 불가로 처리.
- **단점:** 이미지 base64를 직접 저장 시 5~10MB 용량 초과로 즉시 오류 발생. 이미지 없이 좌표 정보만 복원되는 반쪽짜리 구현.
- **권장도:** ★☆☆ (이미지 유지 불가)

### 제안 3. 수동 저장/불러오기 버튼 방식 (가장 직관적)
- **저장 위치:** 사용자 로컬 파일 시스템 (`.json` 사용자 디렉터리에 다운로드)
- **동작 방식:** 상단 툴바에 **[💾 저장]** 및 **[📂 불러오기]** 버튼을 추가. 저장 버튼 클릭 시 모든 `canvasItems` 상태를 JSON 파일(base64 포함)로 다운로드. 불러오기 버튼으로 해당 파일을 다시 올리면 세션을 완벽 복원.
- **특장점:** 브라우저 저장소 용량 무제한. 특정 세션 상태를 파일로 관리하고 팀에 공유하거나 백업 가능.
- **권장도:** ★★☆ (직관적, 명시적 사용자 액션 필요)
